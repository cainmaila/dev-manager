# Dev Manager

English version: [README.md](README.md)

專案位置：https://github.com/cainmaila/dev-manager

以工作流程為核心的軟體開發 agent 交付技能包，也可以把它理解成一位軟體經理。

## 使用 npx skills 安裝

使用 Skills CLI 安裝這個技能包：

```bash
npx skills add cainmaila/dev-manager
```

常見變體：

```bash
# 直接用 GitHub URL 安裝
npx skills add https://github.com/cainmaila/dev-manager

# 安裝為全域技能，而不是只限目前專案
npx skills add cainmaila/dev-manager -g
```

## 使用範例：從需求願望開始

Dev Manager 的設計重點之一，就是你不需要先準備好完整規格，只有一個產品願望也可以開始。

你可以在 agent 對話裡這樣開場：

```text
我想做一個內部工具，可以把會議記錄整理成待辦事項、指派負責人，並每天把摘要送到 Slack。
請用 Dev Manager 幫我從需求一路做到實作。
```

接下來通常會發生的事：

1. Dev Manager 先幫你釐清需求與驗收標準
2. 把你的回答整理成可存檔的需求文件
3. 用規劃技能生成規格、任務拆解與執行計畫
4. 把實作拆成範圍明確、彼此隔離的任務代理
5. 在宣稱完成前，檢查審查結果、整合結果與實際執行情況

前期對話通常會長這樣：

```text
你：我想做一個內部工具，把會議記錄整理成待辦事項。

Agent：第一版是只有你的團隊會使用，還是會擴大到多個部門？

你：先只有我的團隊使用。

Agent：第一版要先支援貼上文字就好，還是也要支援上傳檔案？

你：先貼文字就好，檔案上傳之後再做。

Agent：了解。那第一版我會先把範圍收斂成文字貼上、待辦抽取、負責人指派，以及 Slack 每日摘要。
```

當需求穩定後，你只要把需求文件存到磁碟，流程就會往下走。

例如：

```text
Agent：需求已完整。請將需求文件存檔後提供路徑，例如 ./requirements.md。

你：我存好了，路徑是 ./requirements.md
```

接著 Dev Manager 就會進入任務規劃、執行與驗證，而不是直接跳進一大段難以控制的寫碼流程。

這個倉庫不是為了讓 agent 只把程式碼寫得更快，而是讓它更像一套真正能交付軟體的開發流程。它提供一組技能，讓 agent 從需求釐清、技術規劃、任務切分、隔離實作、審查，到最終驗證，形成一條更完整的交付路徑。

它的目標不是讓 agent 在單次對話裡看起來很厲害，而是讓它在真實專案裡更可控、更可驗證、更值得信任。

## 為什麼會有這個倉庫

現在多數開發 agent 的問題，不是寫不出程式，而是太容易太快開始寫。

真正困難的地方，是把一個模糊想法穩定地變成可規劃、可審查、可測試、可交付的軟體。很多專案失控，並不是因為模型不會產生程式碼，而是因為整個開發流程太鬆散。

常見失敗情境包括：

- 需求還沒穩定就直接開始實作
- 一個大 prompt 一次改太多，最後難以理解與審查
- 多個任務同時進行時互相踩到彼此的變更
- 所謂完成，只代表「程式碼寫出來了」，不代表真的驗證過
- 測試做得太晚、太少，甚至被跳過
- 在對話裡看起來合理，但實際跑起來就失敗

Dev Manager 要解決的，就是這一類問題。

## 它和一般開發 agent 有什麼不同

Dev Manager 不把軟體開發視為一段長 prompt，而是把它視為一條有階段、有交接、有證據的交付流程。

| 面向     | 一般市面上的開發 agent   | Dev Manager                      |
| -------- | ------------------------ | -------------------------------- |
| 主要目標 | 快速產生程式碼           | 從想法一路編排到可驗證交付       |
| 起手方式 | 直接進入實作             | 先釐清需求與驗收標準             |
| 任務切分 | 任務邊界常常模糊         | 任務小而清楚，可獨立驗證         |
| 平行開發 | 共用上下文，容易互相干擾 | 強調任務隔離、明確交接           |
| 審查方式 | 可有可無，或只靠人工補救 | 有明確的範圍與品質審查關卡       |
| 驗證方式 | 常停留在局部測試         | 強調任務、整合、執行層級的證據   |
| 完成定義 | 看起來差不多完成         | 必須有文件、測試、審查與驗證支持 |

## 它解決什麼問題

如果你對下面這些情境有感，這個倉庫就是為你準備的：

- 「我只有一個想法，但 agent 太快開始寫，方向很容易歪掉。」
- 「結果看起來能用，但我不知道到底有沒有真的做完。」
- 「一次改太多檔案，後面根本不容易檢查。」
- 「平行處理多個任務時，常常彼此衝突或互相污染。」
- 「agent 說 done，但實際啟動系統還是出錯。」
- 「我需要的是可檢查的交付證據，不是很有自信的回覆。」

## 核心流程

這個倉庫圍繞一條務實的軟體交付流程設計：

1. 蒐集並澄清需求
2. 把需求轉成技術規格與任務圖
3. 建立執行紀錄與環境就緒關卡
4. 派發範圍清楚的實作任務給專注的 agent
5. 對每個任務做規格符合性與程式品質審查
6. 驗證任務之間的整合是否成立
7. 在宣稱完成前，驗證系統實際執行或部署行為

這套流程的目的不是增加形式，而是降低開發失控的機率。

## 主要技能

這個倉庫的核心技能包含：

- `requirements-interviewer`：把模糊需求整理成明確需求與驗收標準
- `dev-task-planner`：把需求轉成系統規格、任務拆解與可執行工作項目
- `dev-manager`：用非寫碼主管的角色編排整條交付流程
- `senior-engineer`：以測試優先與明確邊界執行單一任務
- `deployment-verifier`：確認最後的系統是否真的能啟動並正確運作
- `dev-doc-cleaner`：審查並清理指定專案根目錄下過時的 dev-manager 文件

這些技能組合起來的目標，是讓 agent 開發流程更穩、更可審核，也更接近真實專案需要的交付方式。

## dev-doc-cleaner

**使用時機：** 專案跑了一段時間後，`TASKS.md`、CR 文件或 `MANAGER_STATE.md` 已出現大量過時紀錄、互相衝突的規劃文件，或已完成的任務模組堆積在目錄裡沒有整理時。

**呼叫方式：**

```text
/dev-doc-cleaner ./my-project
```

**執行流程：**

1. 掃描指定專案根目錄下所有 `dev-manager` 與 `dev-change-manager` 文件
2. 將每份文件分類為 Current（保留）、Stale（待處理）、Obsolete（封存候選）、Conflict（需解決）
3. 列出清理計畫，讓你確認——確認前不動任何檔案
4. 執行：修復衝突 → 壓縮 `TASKS.md` → 封存過時文件 → 僅刪除你明確核准的項目

**安全性：** 確認前全程唯讀。預設封存而非刪除。

## 適合誰使用

Dev Manager 特別適合這幾種人：

- 想保有開發速度，但不想放掉工程紀律的個人開發者
- 從模糊產品想法出發，希望有更穩定落地流程的 builder
- 想嘗試多 agent 開發，但不想讓上下文和檔案變更失控的開發者
- 想要更清楚任務邊界、檢查點與驗證流程的小型團隊

## 設計原則

- 流程優先於即興發揮
- 證據優先於信心
- 小任務優先於超大 prompt
- 隔離優先於上下文污染
- 交付優先於展示

## 如何開始

這個倉庫是一組 skill 資產，不是一個獨立應用程式。

如果你想開始使用：

1. 從 `skills/` 目錄讀核心技能
2. 想跑完整交付流程時，從 `dev-manager` 開始
3. 把 `senior-engineer` 當成任務執行者，而不是專案總控
4. 依照你使用的 agent 平台與工作方式調整這些技能

如果你要的不是「讓 agent 幫你多寫一點 code」，而是「讓 agent 更像一個能把軟體交付完成的開發流程」，那這個倉庫就是朝這個方向打造的。
